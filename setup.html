STEP 01

Copy the Google Sheet template

Required · ~2 min

This Sheet is your card database — it's where every Character, Weapon, Armoury item, Special Order, and Keyword definition lives. The app reads it every time it loads. Making your own copy means you fully own and control it: you can add cards, fix typos, or update stats at any time, and the app picks up the changes immediately on the next refresh.

  1. Open the template using the link below.
  2. Go to File → Make a copy, and save it to your own Google Drive.
  3. Rename it whatever you like — the name doesn't matter to the app.
Open the Free Sheet Template

The Sheet has three tabs along the bottom: List (all your cards — Characters, Weapons, Armoury items, Special Orders, and Rules all live here, distinguished by the Type column), Keywords (the definitions that appear on hover inside the app), and Translations (the English and French UI text — touch this only if you want to add a new language or customise the wording). The template comes pre-filled with a few example rows so you can see exactly how a complete entry looks before you start adding your own.

Understanding the Sheet

Reference

You don't need to fill out the whole Sheet before the app works — the example rows are enough to test the connection in Step 5. But you'll need to understand the structure before you start adding real cards. This section explains how every part of it works.

The List tab: one row = one card

Each row in the List tab represents one card in the game. The Type column tells the app what kind of card it is — Character, Armoury, Special Orders, Command Upgrades, Fireteam Upgrades, or Rules. Every other column is the data that shows up on that card in the app.

Columns explained:

  • Item Name — the card's display name. This is what appears in the browse list and on the card itself. See the "Links in Item Name" section below for how to attach an image.
  • Faction — which faction(s) can use this card. Use comma-separated values for items available to multiple factions (e.g. UNSC, Banished). The app uses this for competitive faction validation.
  • Detachment — the detachment within a faction (e.g. ODST, Jiralhanae). Leave blank for cards with no detachment restriction.
  • Type — see above. Must be one of the exact values the app recognises: Character, Armoury, Special Orders, Command Upgrades, Fireteam Upgrades, Rules.
  • Points — the card's point cost. Numbers only, no units.
  • Keywords — a comma-separated list of keyword names, exactly as they appear in the Keywords tab (e.g. Energy Shield (2), Tactician (1)). The app matches these against the Keywords tab to show hover definitions. Extra text in parentheses like (2) is fine — the match is flexible.
  • RA / FI / SV / Speed / Armour / HP — the six stat columns for Character rows. Leave these blank (or with a dash) for non-Character rows like Armoury items.
  • Weapon 1–3 columns — each weapon has a Type (Ranged,Close Combat, or Grenade), Name, Range, AP, and Keywords sub-column. For a character's built-in weapons, fill these in. For Armoury weapon rows, the app reads the weapon data from the same columns. See "Links in Weapon Names" below for attaching weapon images.
  • Description — free text shown on the card and in the info panel. For Armoury items and Special Orders, this is the rules text. For Rules rows, this is the full rule text itself. Supports plain text; line breaks are fine.
  • Restrictions — controls when an item can be added to a roster. Leave blank if there are no restrictions. See the Restrictions Guide tab in your Sheet for the full token reference.

Adding image links — Item Name and Weapon Name cells

The app can display card art and weapon images by reading a link embedded directly in the cell — not from a separate column. Here's how it works:

  1. Upload your card images to Google Drive (any folder is fine).
  2. Right-click the image in Drive → Get link → change sharing to "Anyone with the link" → copy the link.
  3. In your Sheet, click on the Item Name cell (or the Weapon Name cell) for that card.
  4. Press Ctrl+K (or Cmd+K on Mac) to open the Insert Link dialog.
  5. Paste the Google Drive link and click Apply. The cell text stays the same — it just now has a link underneath it that the app can read.

The app checks whether a cell has a link attached to it. If it does, it uses that link as the image source. If it doesn't, it shows the default card placeholder. This means you can have a name like MA40 Assault Rifle display as text in the Sheet, but open as a card image inside the app.

Google Drive sharing: "Anyone with the link"
The image link needs to be set to "Anyone with the link can view" — otherwise the app will get a permission error when it tries to load the image. The image is only displayed inside your own hosted copy of the app; this setting doesn't make the image searchable or publicly listed anywhere.

The Keywords tab

The Keywords tab has two columns: Keyword (the name, exactly as it'll be referenced in the List tab's Keywords column) and Definition (the rules text that appears on hover inside the app).

  • A keyword in the List tab matches an entry here if the keyword name contains the entry's name — so Energy Shield (2) in a card's keywords will match the Energy Shield row in this tab.
  • If a keyword in a card's row has no matching entry here, the hover tooltip just won't appear for that keyword. Nothing breaks.
  • Keep definitions up to date with any official errata — this is the single source of truth the app uses for keyword text, so if Mantic changes a keyword's rules, just edit the definition here and every card that uses it updates automatically.
  • There's no limit on definition length, but shorter is more readable in the hover tooltip.

The Translations tab

The Translations tab controls every piece of text in the app's UI — button labels, error messages, modal titles, PDF export strings, everything. It ships with English and French columns already filled in. You don't need to touch this tab unless you want to:

  • Add a new language: create a new column to the right of French. The column header becomes the language name shown in Settings. Fill in every row in that column with your translated text. The app will automatically make this language available as a choice in the Settings → Language selector — no code change required.
  • Customise existing text: just edit the English or French cell directly. For example, if you want to change "Add to Roster" to "Add to List" in your own copy, edit that cell and the app picks it up on the next refresh.

The Key column (first column, locked) is what the app uses internally to find each string. Don't edit the Key column. The English and French values are what actually appear in the UI.

Tip: use the Sheet for everything text-related
Because all UI text comes from your Sheet, you can translate the entire interface into any language without touching a line of code. The same goes for rule updates: if Mantic errata a keyword, edit the Keywords tab and you're done. No app update needed.

The Restrictions column — quick reference

The Restrictions column controls when an item can be added to a roster, and for Special Orders, which competitive detachment context they appear in. Leave it blank if there are no restrictions. Multiple tokens are separated by | — all conditions must be true.

Character & roster conditions — who must be in the roster for this item to be available:

  • REQUIRES_CHAR_NAME_INCLUDES:value — roster must include a character whose name contains this text (e.g. REQUIRES_CHAR_NAME_INCLUDES:master chief)
  • REQUIRES_CHAR_FACTION_INCLUDES:value — roster must include a character of this faction
  • REQUIRES_CHAR_DETACHMENT_INCLUDES:value — roster must include a character in this detachment
  • ALL_CHARS_DETACHMENT:value — every character in the roster must be this detachment
  • ALL_CHARS_FACTION:value — every character must be this faction

Special Orders — competitive detachment visibility — use these instead of the Detachment column for Special Orders. The Detachment column on Special Order rows must stay blank (writing anything there pollutes the browse filter dropdown):

  • ALL_UNITS_ONLY — this Special Order only appears in comp when "All Units" is selected; hidden for specific detachments like ODST or Jiralhanae
  • DETACHMENT_ONLY:value — this Special Order only appears in comp for this specific detachment (e.g. DETACHMENT_ONLY:ODST). Use multiple pipe-separated entries to allow more than one detachment: DETACHMENT_ONLY:ODST|DETACHMENT_ONLY:Jiralhanae

Weapon-slot rules:

  • REQUIRES_WEAPON:value — character must already have this weapon on their base card (used for replacement weapons like combat knives)
  • BLOCKS_LEGEND — item cannot be equipped by Legend-tier characters
  • EXEMPT_FROM_WEAPON_LIMIT — item doesn't count against the weapon-per-character cap

The full reference with worked examples is in the Restrictions Guide tab of your Sheet.

STEP 02

Deploy the Apps Script

Required · ~5 min

Apps Script is a small free Google service that connects your Sheet to the app. Think of it as a messenger: the app asks it "give me the card data from this Sheet," it reads the Sheet and sends the data back. The connector script (Code.gs) is already included inside your copied Google Sheet — you don't need to download or paste anything for that part.

  1. In your copied Sheet, go to Extensions → Apps Script. A new tab will open with a code editor. The script is already there — don't delete or edit it.
  2. Click Deploy → New deployment.
  3. Next to "Select type," click the gear icon and choose Web app.
  4. Set Execute as to Me, and Who has access to Anyone.
  5. Click Deploy. Google will ask you to authorise the script — click through the prompts and allow it.
  6. Copy the Web app URL shown on the final screen. It ends in /exec. Save this somewhere — you'll paste it into the app in Step 5.
Optional: pre-configure script permissions with appsscript.json
Copying the appsscript.json manifest into your Apps Script project before deploying sets the correct OAuth scopes in advance, which makes the authorisation step smoother and avoids unexpected permission prompts. To use it: in the Apps Script editor, go to Project Settings → Show "appsscript.json" manifest file in editor, then replace the file's contents by pasting the text below. Click the button to copy it to your clipboard.

The "Anyone" setting — and why it's safe
"Who has access: Anyone" sounds alarming, but it controls only who can call the script's web address — not who can see or edit your Sheet. The URL itself is long and effectively unguessable, and the script only exposes the specific read-only actions written into it. There is no way to reach anything else in your Google account through it. This is the standard way Apps Script web apps work. See the FAQ for a longer explanation.
Most common mistake here
If "Who has access" isn't set to Anyone, the app will fail to connect later with a confusing error. Double-check this setting before clicking Deploy.

STEP 04

Host it with GitHub + Cloudflare

Required · ~7 min

The app needs to be hosted at a real web address — not opened as a file on your computer — because it communicates with your Google Sheet over the internet. GitHub stores your files; Cloudflare turns them into a live website. Both are free for this use.

Part A — GitHub:

  1. Create a free account at github.com if you don't have one already.
  2. Click the + button in the top right → New repository.
  3. Give it any name (e.g. my-flashpoint-list) and set the visibility to Private. See the callout below for why.
  4. Click Create repository.
  5. Once it's created, click Add file → Upload files.
  6. Drag all five files from your purchase — index.html, sw.js, manifest.webmanifest, apple-touch-icon.png, and wrangler.json — into the upload area. All five must be in the same folder.
  7. Click Commit changes at the bottom.

Part B — Cloudflare:

  1. Create a free account at cloudflare.com.
  2. From your dashboard, go to Workers & Pages → Create → Pages → Connect to Git.
  3. Authorise Cloudflare to access your GitHub account when prompted.
  4. Select the repository you just created.
  5. On the build settings screen, leave everything as the default — no build command, no output directory. These are finished static files, not a project that needs compiling.
  6. Click Save and Deploy. After a minute or two, Cloudflare will show your live URL.
  7. Open that URL in your browser to confirm the app loads. You'll see the first-run setup screen — that's the next step.
Why Private, not Public
Keep the repository Private. The app reads card data from your own Google Sheet, but that data — names, stats, rules text — comes from Mantic's published game. A private repository keeps that content from being re-published in a browsable form on GitHub. Cloudflare deploys from private repositories just as easily as public ones, so this costs you nothing extra.
All five files, same folder
All five files must be uploaded together in the same folder: index.html, sw.js, manifest.webmanifest, apple-touch-icon.png, and wrangler.json. The wrangler.json file tells Cloudflare how to handle the deployment correctly. Without the other companion files, "Add to Home Screen" won't show the correct name or icon and won't open full-screen on mobile — iOS specifically requires apple-touch-icon.png for the home screen icon.

STEP 05

First-run setup wizard

Required · ~2 min

Open your new live URL from Step 4. Because it's a fresh copy with no connection configured yet, the app shows a setup screen before anything else. This is where you paste in the Apps Script URL from Step 2 and test the connection.

  1. Paste the Web app URL you saved at the end of Step 2 into the input field.
  2. Click Test Connection. If everything is connected correctly, you'll see a green confirmation showing how many rows of card data it found in your Sheet.
  3. Click Save & Continue.

The app is now connected to your Sheet. From here, anything you add or edit in the Sheet will appear in the app after a page refresh.

If Test Connection fails
The most common cause is the "Who has access" setting from Step 2 not being set to Anyone. Go back to Extensions → Apps Script → Manage deployments, check the setting, and redeploy if needed. The new URL after redeployment will be different — update it in the wizard.
STEP 06

Connect Supabase for cloud saves

Optional · ~5 min

This step adds cloud roster saves — the ability to save a roster in the app and load it again on any device (your phone, your partner's tablet, a different computer). If you skip it, you can still save and share rosters as JSON files, which works with zero setup and never expires.

  1. Create a free account at supabase.com.
  2. Click New project. Give it a name, set a database password (store it somewhere — you won't need it often, but Supabase requires it), and choose the region closest to you.
  3. Wait for the project to finish initialising (usually 1–2 minutes).
  4. Go to the SQL Editor tab in your project dashboard.
  5. Download the free supabase-setup.sql file using the link below, copy its contents, and paste it into the editor. Click Run. This creates the single table the app needs to save rosters.
Download supabase-setup.sql — Free
  1. Go to Project Settings → API. Copy the Project URL and the anon public key (not the service_role key).
  2. Back in the app, open Settings → Edit Connections, paste both values into the Supabase fields, and click Test Connection.

STEP 07

Run a Health Check

Recommended · ~1 min

Before building your first list, go to Settings → Sheet Health Check → Run Health Check. This scans your card data for the most common setup problems: missing required fields, duplicate card names, keywords referenced in cards that don't exist in the Keywords tab, and broken image links.

A clean result (zero errors) means your Sheet is correctly structured and the app will behave as expected. If there are warnings, the Health Check tells you exactly which row and column to look at.

BONUS

Add to Home Screen on iPhone or iPad

Optional · ~1 min

Once the app is live at your Cloudflare URL, you can install it on your iPhone or iPad so it opens like a native app — full screen, no browser bar, with its own icon on your home screen. This works in Safari only; Chrome and Firefox on iOS can't install web apps to the home screen.

  1. Open your Cloudflare URL in Safari on your iPhone or iPad.
  2. Wait for the page to fully load — you should see the app's first-run setup screen or the Browse panel if you've already connected your Sheet.
  3. Tap the Share button at the bottom of the screen (the box with an arrow pointing up).
  4. Scroll down in the share sheet and tap Add to Home Screen.
  5. Give it a name (or keep the default) and tap Add in the top right.
  6. The app now appears on your home screen with its own icon. Opening it from there launches it full-screen.
Must be Safari — not Chrome or Firefox
"Add to Home Screen" for web apps only works in Safari on iOS. If you open the Cloudflare URL in Chrome or Firefox on your iPhone, the option will either be missing or won't create a proper installed app experience. Once installed, you can use it in any orientation and it opens without the browser's address bar.
Offline use after install
After installing and loading the app once with signal, it'll keep working for the rest of that session even if you lose connection. The apple-touch-icon.png file in your GitHub repository is what gives the home screen icon its correct appearance on iOS — without it, iOS falls back to a generic screenshot thumbnail.
